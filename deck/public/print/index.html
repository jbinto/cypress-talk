What drove us to Cypress?

    Component tests with `enzyme` shallow rendering are fragile

  • 🗑 Tests weren't useful for verifying dependency upgrades
  • 😰 Refactoring meant re-writing most or all of the tests
  • 💰 Which means our tests didn't pay for themselves

    Test "like a user" to enable refactoring

    Our mistake was testing implementation details.

    • 🚧 The internals of React were entirely rewritten between React 15 and React 16
    • 👩‍🔬 They only tested their public APIs
      • Tests didn't need to be rewritten
      • And they actually verified things still worked
    • 💻 For a web application, the user interface is the public API
    • ⏭

    Cypress is designed for reliability

  • 🕴 It hides the complexities of async
  • 🚫 You can always* write deterministic, flake-free tests
  • 👨‍⚖️ (* if you follow the rules)
  • ⏭

    • Cypress has great documentation

    • 🔤 It's simple (sometimes)
    • 👮‍♂️ But in order to guarantee reliability, you need to follow rules
    • 👨‍🎓 You may need to unlearn the way you've tested in other environments
    • 📚 Read the docs to learn more!
    • ⏭

    The tests themselves

  • 🧙‍♂️ Stub out REST API calls with cy.route()
  • 🏗️ Stub out browser APIs (or your own app's APIs!) with cy.stub()
  • 🕰 Fast-forward in-app timers with cy.tick()
  • ⚙️ Write arbitrary tasks in Node with cy.task()
  • 🕸 Make web requests with cy.request()
  • ⏭

    • The test runner

  • ⌚️ You can view snapshots, "before"/"after" of each test step
  • 🔓 You can access the console, and interact with your app
  • 🎥 You can record screenshots and videos of test runs
  • 🧾 You can get code coverage metrics
